## 一、设计概述

### 1.1 设计目标

将原 v1.0 纯内存版本升级为支持数据持久化的 v2.0 版本，解决以下核心问题：

| 问题 | v1.0 方案 | v2.0 数据库方案 |
|------|----------|----------------|
| 对话历史丢失 | SessionState，刷新即清 | SQLite持久化存储 |
| 无用户区分 | 全局共享一个API Key | 独立用户表+bcrypt密码 |
| API Key暴露 | 硬编码在前端 | AES加密存储于数据库 |
| 知识库检索低效 | 字符串`in`匹配 | 结构化存储+多字段检索 |
| 分析结果无法保存 | 无存储机制 | 报告归档表 |

### 1.2 技术选型

| 技术 | 版本 | 用途 |
|------|------|------|
| SQLite | 3.39+ | 嵌入式关系型数据库，零配置 |
| SQLAlchemy | 2.0+ | Python ORM，面向对象操作数据库 |
| bcrypt | 4.0+ | 密码哈希（单向加密） |
| cryptography | 41.0+ | AES对称加密（API Key加密存储） |

**选型理由**：
- **SQLite**：无需独立服务器进程，单个文件即可运行，适合课程作业场景
- **SQLAlchemy**：提供ORM封装，业务代码无需手写SQL，便于维护
- **bcrypt**：行业标准的密码哈希算法，自动加盐防彩虹表攻击
- **cryptography**：Python官方推荐的加密库，AES-256-CBC模式

## 四、安全设计

### 4.1 密码安全

```
注册流程：
用户输入密码 → bcrypt哈希(自动加盐) → 存储password_hash

登录流程：
用户输入密码 → bcrypt校验(密码 + hash) → 返回True/False

特点：
- 单向哈希，无法从hash反推原密码
- 自动加盐，相同密码的hash值不同
- 可调节强度因子(cost=10)
```

### 4.2 API Key安全

```
存储流程：
用户API Key → AES-256-CBC加密 → base64编码 → 存入数据库

使用流程：
数据库读取 → base64解码 → AES解密 → 明文API Key → 调用Kimi API

加密参数：
- 算法：AES-256-CBC
- 密钥：32字节固定密钥（课程演示用）
- IV：16字节固定IV（课程演示用）
- 生产环境建议：密钥通过环境变量传入，IV随机生成并存储
```

## 五、性能优化

### 5.1 索引策略

| 表名 | 索引字段 | 索引类型 | 理由 |
|------|---------|---------|------|
| users | username | UNIQUE | 登录查询、防重复注册 |
| conversations | user_id | 普通索引 | 查询用户对话列表 |
| messages | conversation_id | 普通索引 | 查询对话历史消息 |
| knowledge_entries | category | 普通索引 | 分类浏览查询 |

### 5.2 查询优化

- 对话列表按 `is_pinned DESC, updated_at DESC` 排序（置顶优先+最近优先）
- 消息查询按 `timestamp ASC` 排序（保证时序正确）
- 知识库搜索使用多字段 `OR` 条件（title/content/keywords）
